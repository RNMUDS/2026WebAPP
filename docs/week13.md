## 解説

### 1. 総合制作の目標

第1〜12回で学んだ技術を組み合わせて、**複数の利用者が同時に参加できるインタラクティブな3D Webアプリケーション**を制作する。

**使える技術:**
- Express.js（サーバ、ルーティング、API）
- A-Frame（3D空間��モデル、アニメーション）
- Socket.IO（リアルタイム通信、アバタ同期）
- SQLite（データの永続化）

### 2. 作品テーマの例

| テーマ | 概要 |
|--------|------|
| バーチャル教室 | 3D教室で生徒がアバタで入室、チャットや挙手ができる |
| 共同アート空間 | 3D空間��オブジェクトを配置してアート作品を共同制作 |
| 宝探しゲーム | 3D空間に隠された宝を複数人で探すゲーム |
| バーチャ���展示会 | 自分の作品を3D空間に展示し、来場者がコメントできる |
| ミニ対戦ゲーム | 簡単なルールの3Dマルチプレイゲーム |

### 3. 設計の進め方

#### ステップ1: コンセプトを決める

- **何を作るか**（1文で説明できるか？）
- **誰が使うか**（利用者のイメージ）
- **面白さ・価値は何か**（なぜこれを作るのか）

#### ステップ2: 機能一覧

必要な機能を洗い出す：

| 機能 | 優先度 | 技術 |
|------|--------|------|
| 3D空間の表示 | 必須 | A-Frame |
| アバタの��期 | 必須 | Socket.IO |
| チャット | 必須 | Socket.IO |
| オブジェクト操作 | 必須 | A-Frame + Socket.IO |
| データ保存 | あれば | SQLite |
| スコア表示 | あれば | Express API |

**優先順位を付ける:**
- 必須: これがないと成り立たない機能
- あれば: 時間があれば追加する機能

#### ステップ4: データ設計

サーバで管理するデータを決める：

```
users: {
  socket.id: { id, color, position, rotation, name }
}

objects: {
  "obj-1": { color, position, rotation, state }
}

messages: [
  { name, text, time }
]
```

DBに永続化するデータ：
- objects テーブル（位置、色、状態）
- messages テーブル（チャットログ）
