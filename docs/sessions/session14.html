1.1 最終制作の要件

読み方: requirement = リクワイアメント（「要件」「必須条件」の意。 require「必要とする」＋ -ment「名詞化」）。 派生語の MVP＝エムブイピー（Minimum Viable Product = 最小限の動作する製品）、use case＝ユースケース（システムの利用シーン）、architecture＝アーキテクチャ（ギリシャ語 arkhi-「主要な」＋ tekton「建築家」= システム全体の構造設計）。

家を建てる前に設計図を描くように、アプリを作る前に何を作るかを明確にしておくことが重要です。
設計なしに作り始めると、途中で方向がわからなくなり、完成しないリスクが高まります。

最終制作では、これまで学んだ技術を組み合わせてマルチユーザ3D Webアプリケーションを作ります。

使用技術（必須）

技術用途学んだ回
A-Frame 1.7.13D空間の構築第4〜8回
Node.js + Expressサーバの構築第2〜3回
Socket.IOリアルタイム通信第9〜12回

基本機能（必須） 設計課題に含む

#機能説明
1マルチプレイ（アバター同期）複数のブラウザからアクセスし、各ユーザのアバターが3D空間内に表示される。位置・回転がリアルタイムに同期される。
2吹き出しチャット3D空間内でテキストメッセージを送信でき、アバターの上に吹き出しとして表示される。

自由要素（個性を出す部分）

  • テーマ ── 空間のコンセプト（教室、宇宙、水中、美術館、ゲーム空間など）
  • 3Dデザイン ── オブジェクトの配置、色、テクスチャ、ライティング
  • 追加機能 ── オブジェクト操作、エフェクト、音声、スコア、DB永続化など

💡 具体例 ── 要件の優先順位

例えば「VR水族館」を作る場合、優先順位は次の通り:

  1. 必須: ダイバーアバターの同期、泡形の吹き出しチャット（これがないと「マルチユーザ3D」にならない）
  2. 追加1: 魚の群れアニメーション（テーマの説得力を上げる）
  3. 追加2: チャット履歴の保存（実装難度が低く効果が大きい）
  4. 余力: BGM、ライティング演出（時間が余ったら）

→ 「やりたい順」ではなく「無いと困る順」で並べると、途中で時間切れになっても最低限動くアプリが残る。

❓ デバッグでよくある疑問

よくある質問答え
エラーが出ているのにどこを直せばいいか分からない エラーメッセージの1行目とファイル名:行番号を読む。原因は大抵その位置か、その近くで定義した変数にある。
console.log を入れたのに何も表示されない そもそもその行に処理が到達していない。console.log('A'), console.log('B') のように複数入れて、どこまで実行されるか確認する。
2タブで試したらアバターが2つ表示される（自分のも見える） サーバが io.emit を使っている。socket.broadcast.emit に変えると「送信者以外」に配信される。
「動いていたコード」が動かなくなった git diff で直近の変更を確認するか、最後に動いた時点との差分を見る。「動く状態に戻してから1つずつ追加」が最速。

💡 具体例: エラーから原因にたどり着く流れ

ブラウザに次のエラーが表示されたとします: 

Console
Uncaught TypeError: Cannot read properties of undefined (reading 'position')
    at app.js:42:18

読み解き方:

  1. app.js の42行目で問題が起きている
  2. undefined.position を読もうとしている → 何かのオブジェクトが undefined
  3. 該当行を見ると data.position と書いてある → data が undefined
  4. 送信側で socket.emit('move', posData) を確認 → posData がそもそも空だった
  5. 修正: 送信前に if (posData) { ... } を入れる or 送信ロジックを直す

→ 「エラーメッセージから5分以内に原因の行を特定する」のがデバッグの基本リズム。

エラー1: CORS エラー

エラーメッセージ:
Access to XMLHttpRequest at 'http://localhost:3000' from origin 'http://127.0.0.1:5500' has been blocked by CORS policy

原因: Live Server（ポート5500）からNode.jsサーバ（ポート3000）にアクセスしている。異なるオリジン間の通信はブラウザがブロックする。

解決方法:

  • Live Serverを使わず、Node.jsサーバから直接HTMLを配信する
  • express.static() で public フォルダを公開し、http://localhost:3000 でアクセスする
JavaScript ── server.js
const express = require('express');
const app = express();

// publicフォルダの中身を静的ファイルとして配信
app.use(express.static('public'));

// http://localhost:3000 でindex.htmlが表示される

エラー2: Socket.IO 接続失敗

エラーメッセージ:
GET http://localhost:3000/socket.io/?... 404 (Not Found)

原因: Socket.IOがサーバ側で正しく設定されていない。Expressだけ起動して Socket.IO をサーバに紐づけていない（const io = socketIo(server) を書いていない）。

解決方法: 第10回・第11回と同じく、app.listen() が返すサーバ本体を socketIo() に渡して Socket.IO を紐づける。

JavaScript ── server.js
const express = require('express');
const socketIo = require('socket.io');

const app = express();

エラー3: 3Dモデルが表示されない

症状: glTFモデルの読み込みを指定しているが、3D空間に何も表示されない。

よくある原因と解決方法:

原因解決方法
ファイルパスが間違っている相対パスを確認。public/models/scene.glb なら src="models/scene.glb"
モデルのスケールが極端scale="0.01 0.01 0.01" のように調整する
モデルの位置がカメラから遠いposition="0 0 -5" のように手前に配置する
MIME typeエラー.glb ファイルが application/octet-stream で配信されているか確認

デバッグのコツ: まず<a-box>で同じ位置にテスト表示し、位置が正しいか確認してからモデルに置き換える。

エラー4: アバターが同期されない

症状: 他のユーザのアバターが表示されない、または動かない。

確認ポイント:

  • console.log でSocket.IOのイベントが発火しているか確認する
  • サーバ側で socket.broadcast.emit を使っているか（io.emit だと自分にも送信される）
  • クライアント側で受信したデータを正しくA-Frameエンティティに反映しているか
  • 位置データのフォーマットが送信側と受信側で一致しているか
JavaScript ── デバッグ用console.log
// クライアント側: 送信時に確認
socket.emit('move', posData);
console.log('送信:', posData);

// クライアント側: 受信時に確認
socket.on('moved', (data) => {
  console.log('受信:', data);
  // ここでA-Frameエンティティを更新
});

やってみよう Step 1「テーマ案の比較」 解答例

3つのテーマ案を比較したサンプル:

テーマ案必須機能をどう組み込めるか実現難度
① バーチャル教室生徒アバターが座席に集まり、吹き出しで質疑応答する低（家具配置がシンプル）
② 宇宙ステーション宇宙服アバターが浮遊し、無線通信風の吹き出しで会話する中（重力なしの演出が必要）
③ 美術館来館者アバターが展示前で立ち止まり、感想を吹き出しで共有する中（展示物の3Dアセットが必要）

選ぶ基準: 自分が90分で実装できそうか、3Dアセットが手元にあるか、テーマに思い入れがあるか。① バーチャル教室のように基本図形（box / cylinder / plane）だけで形が成立するテーマが最も実現しやすい。

→ 設計ワークシートの途中であれば変更OK。ただし、グループ内発表より前に確定させること。実装が始まってからの変更は時間切れリスクが高いので推奨しない。

Q2. 追加機能はいくつまで設計すべき？

→ 標準課題は1つ以上。発展課題は1つを詳細に設計するほうが、複数を浅く設計するより評価が高い。「実装方法・ユーザ価値」まで掘り下げられる粒度を選ぶ。

Q3. スケッチは絵が下手でも大丈夫？

→ 問題ない。設計レビューで伝わればよい。位置関係・サイズ感・UI配置の3点が読み取れることが重要。きれいに描く必要はなく、矢印やラベルで補足してOK。

Q4. 過去の回で学んだ技術を全部使わないとダメ？

→ 必須はA-Frame / Node.js+Express / Socket.IO のみ。SQLite やアニメーションなどは追加機能で必要になったら使う。使わない技術があってOK。

Q5. Socket.IO イベントの名前ルールは？

→ 動詞形（join, move, chat）または「主語-動詞」形（player-joined, chat-broadcast）。クライアント→サーバは命令形、サーバ→全クライアントは過去形/結果形が読みやすい。
