1.1 JSON とは 課題に出ます

友達と連絡先を交換するとき、「名前: 田中、電話: 090-xxxx、趣味: 読書」のように項目名と値のペアで書きます。JSON はコンピュータ同士がデータを交換するときの「共通フォーマット」で、同じようにキーと値のペアで構造化されたデータです。

JSON（JavaScript Object Notation）は、データを構造化して表現するテキスト形式です。Webアプリケーションでのデータ通信で最も広く使われています。

  "hobbies": ["読書", "プログラミング", "カフェ巡り"],
  "university": {
    "name": "○○大学",
    "year": 2
  }
}

1.2 Fetch API（フェッチ エーピーアイ） ── ブラウザからサーバにリクエストを送る 課題に出ます

1.3 async/await 基礎 課題に出ます

レストランで料理を注文すると、料理ができるまで待つ必要があります。でも待っている間、スマホを見たり友達と話したりできますよね？
await もこれと同じです。関数の中ではサーバの返答を「待つ」けど、関数の外の処理はその間も止まらず先に進みます。

fetch() はサーバとの通信を行うため、結果がすぐには返ってきません（非同期処理）。async/await を使うと、非同期処理を同期的（上から順番に実行されるよう）に書けます。

JavaScript
// async 関数の中でのみ await が使える
async function loadData() {
  const response = await fetch('/api/data');   // サーバにリクエスト → 返答を待つ
  const data = await response.json();          // JSONを解析 → 完了を待つ
  console.log(data);                            // データを使う
}
キーワード意味
async関数を非同期関数として宣言する。この関数の中で await が使えるようになる
await非同期処理の完了を待つ。結果が返ってくるまで次の行に進まない

await を使わないと、サーバの返答が届く前に次の処理が進んでしまい、data が undefined（未定義）になります。

async/await を「1個のボール」で理解する（抽象アニメーション）

JavaScript の実行は 1個のボールで表せます。
ボールは1つしかなく、同時に2か所では動けません（シングルスレッド）。
このボールが「関数の外」レーンと「関数の中」レーンをどう移動するかを追いかければ、await の動きが自然に分かります。

💡 ポイント1

ボール（JS実行）は1個だけ。同じ瞬間に2か所では動けない。

💡 ポイント2

await でボールは関数を脱出。関数の外（D）が先に動く。

💡 ポイント3

レスポンスが届いたら、ボールが関数の中に戻って C を実行。

なぜ実行順が A → B → D → C になるのか？

普通のプログラムは上から順番に実行されます。しかし await がある行に来ると、特別なことが起きます。
以下のコードを例に、1行ずつ何が起きるかを追いかけてみましょう。 

JavaScript
console.log('A: fetch開始前');       // ─── 行1

async function getData() {           // ─── 行2（関数を定義）
  console.log('B: fetch実行');       // ─── 行3
  const response = await fetch('/api/items');  // ─── 行4 ★ここがポイント
  const data = await response.json();         // ─── 行5
  console.log('C: データ受信', data);  // ─── 行6
}

getData();                             // ─── 行7（関数を呼び出す）
console.log('D: fetch呼び出し後');   // ─── 行8
順番実行される行何が起きるか
1番目 行1 console.log('A') を実行。普通の処理なのでそのまま出力される。
→ 画面に「A: fetch開始前」と表示
2番目 行7 → 行3 getData() が呼ばれ、関数の中に入る。console.log('B') を実行。
→ 画面に「B: fetch実行」と表示
★ ここで特別なことが起きる（行4）
await fetch('/api/items') に到達。await は「サーバの返答が届くまで待て」という意味。

でも、サーバの返答が届くには何百ミリ秒もかかる。その間ずっと止まっていたら、他の処理ができない。
だから JavaScript は「この関数の中での実行は一旦やめて、関数の外に戻ろう」と判断する。

つまり: await に到達した瞬間、関数 getData() の実行は一時停止され、行8に飛ぶ。
3番目 行8 関数の外に戻り、console.log('D') を実行。サーバの返答はまだ届いていないが、外の処理は待たされない。
→ 画面に「D: fetch呼び出し後」と表示
⏳ 時間が経過... サーバからデータが届く
サーバが処理を終え、JSONデータがネットワークを通じて返ってくる。
JavaScript は「あ、返答が届いた。さっき一時停止した getData() の続きを再開しよう」と判断する。
4番目 行5 → 行6 getData() の続きが再開。response.json() でデータを変換し、console.log('C') を実行。
→ 画面に「C: データ受信 {...}」と表示

たとえ話で理解する

あなたは料理を注文（= await fetch()）しました。
料理ができるまでの間、あなたはスマホを見たり友達と話したり（= console.log('D')）できます。
料理が届いてから、食べ始めます（= console.log('C')）。

注文してから料理が届くまでの間、あなたはじっと座って何もしないわけではない。
これが await の仕組みです。関数の中では待つけど、関数の外の世界は止まらず動き続けます。

最終的な出力順:

A: fetch開始前 → B: fetch実行 → D: fetch呼び出し後 → C: データ受信 {...}

コードの見た目の順番は A→B→C→D だが、実際の実行順は A→B→D→C。await で「待つ」間に D が先に実行されるから。

1.4 GET と POST の使い分け 課題に出ます

GET は図書館で「この本を見せてください」と頼むこと（情報を取得する）。POST は窓口に「この申込書を提出します」と書類を渡すこと（情報を送信する）。

// サーバ側: /api/messages にPOSTリクエストが来たら実行されるルート
app.post('/api/messages', function(req, res) {  // req=リクエスト（送られてきたもの）, res=レスポンス（返すもの）

1.5 REST API（レスト エーピーアイ） の考え方

図書館の本は「ジャンル → 著者名 → タイトル」のように整理されています。REST API も同じで、データをリソース（資源）として整理し、URLで「どのデータにアクセスするか」を表現します。

REST（Representational State Transfer）は、WebAPIの設計思想です。URLでリソースを表し、HTTPメソッドで操作を表します。

REST API は特別なライブラリではない

REST API はURLの設計ルール（考え方）であり、追加のインストールは不要です。Express の app.get(), app.post() など標準の機能だけで作れます。
「/api/users で一覧を返す」「/api/users/1 で1件を返す」のようにURLを整理するだけで REST API になります。

HTTPメソッドURL意味Express のコード
GET/api/usersユーザ一覧を取得app.get('/api/users', ...)
GET/api/users/1ID=1 のユーザを取得app.get('/api/users/:id', ...)
POST/api/users新しいユーザを作成app.post('/api/users', ...)
PUT/api/users/1ID=1 のユーザを更新app.put('/api/users/:id', ...)
DELETE/api/users/1ID=1 のユーザを削除app.delete('/api/users/:id', ...)

この授業では GET と POST を主に使います。URLは「何のデータか」を表し、メソッドは「何をするか」を表すのがポイントです。

1.6 DOM操作でデータを表示する 課題に出ます

ホワイトボードに書かれた内容を消して新しく書き直すように、JavaScriptでHTMLの中身を動的に書き換えることができます。これがDOM操作です。

サーバから取得したデータをページに表示するには、DOM（Document Object Model）を操作してHTMLの内容を更新します。

JavaScript
// HTML側に <div id="result"></div> があるとする

// id="result" の HTML要素を取得する（ページの中から探す）
const resultDiv = document.getElementById('result');
// その要素の中身（HTML）を書き換える → 画面に反映される
resultDiv.innerHTML = '<p>サーバから取得したデータです</p>';
メソッド / プロパティ説明
document.getElementById('id')指定したIDを持つHTML要素を取得する
要素.innerHTML要素の中身のHTMLを読み書きする
要素.textContent要素のテキスト内容を読み書きする（HTMLタグは反映しない）

Fetch API でデータを取得 → DOM操作で表示。この2つを組み合わせることで、ページを再読み込みせずにサーバのデータを画面に反映できます。

例題

Fetch API の処理の流れを読む

以下のコードがどの順番で実行されるかを確認しましょう。

JavaScript
console.log('A: fetch開始前');       // ① 最初に実行される（同期処理）

async function getData() {           // 非同期関数を定義する
  console.log('B: fetch実行');       // ② getData()が呼ばれたらすぐ実行される
  const response = await fetch('/api/items');  // サーバに問い合わせ → ここで「待つ」
  const data = await response.json();         // JSONに変換 → ここでも「待つ」
  console.log('C: データ受信', data);  // ④ サーバから応答が届いてから実行される
}

getData();                             // 関数を呼ぶ（await中は関数の外に戻る）
console.log('D: fetch呼び出し後');   // ③ awaitで待っている間に先に実行される
順番出力理由
1A: fetch開始前同期処理なのでそのまま実行される
2B: fetch実行getData() が呼ばれ、await の前まで実行される
3D: fetch呼び出し後await でサーバ応答を待つ間、次の行が先に実行される
4C: データ受信 ...サーバから応答が届いた後に実行される

ポイント: await で「待つ」のは関数の中だけで、関数の外の処理は先に進みます。

      const msg  = document.getElementById('msgInput').value;

      // サーバの /api/message にPOSTリクエストを送る

JavaScript ── server2.js
const express = require('express');  // Express ライブラリを読み込む
const app = express();              // Express アプリケーションを作成する

// ★重要: POSTで送られたJSONデータを req.body で読めるようにする
// この行がないと req.body は undefined（未定義）になる
app.use(express.json());

// publicフォルダの中身をそのまま配信する（HTMLやCSSなど）
app.use(express.static('public'));

// GET: /api/members にアクセスされたらメンバー一覧をJSONで返す
app.get('/api/members', function(req, res) {
  res.json({                          // JSONデータをレスポンスとして返す
    team: 'Webプログラミングチーム',
    members: [
      { name: '田中', role: 'リーダー' },
      { name: '佐藤', role: 'デザイン' },
      { name: '鈴木', role: 'サーバ担当' },
      { name: '高橋', role: 'フロント担当' }
    ]
  });
});

// POST: /api/message にデータが送られてきたら受け取って応答する
app.post('/api/message', function(req, res) {
  console.log('受信:', req.body);   // 受け取ったデータをターミナルに表示（デバッグ用）
  res.json({                          // クライアントにJSON形式で応答を返す
    reply: req.body.name + 'さん、メッセージ「' + req.body.message + '」を受け取りました！',  // 受け取った名前とメッセージを使って返事を作る
    receivedAt: new Date().toLocaleString('ja-JP')  // 現在の日時を日本語形式で追加する
  });
});

app.listen(3000, function() {      // ポート3000でサーバを起動する
  console.log('サーバ起動: http://localhost:3000');
});

例題 やってみよう解答（console.log('E: 最後') を追加した場合）

正解の出力順: A → B → D → E → C

順番出力理由
1A: fetch開始前同期処理なのでそのまま実行される
2B: fetch実行getData() が呼ばれ、await の前まで実行される
3D: fetch呼び出し後await でサーバ応答を待つ間、関数の外に戻って実行される
4E: 最後D の次の行なので、そのまま続けて実行される（同期処理）
5C: データ受信 ...サーバから応答が届いた後に、getData() の続きが再開して実行される

E は D と同じく「関数の外の処理」なので、await 中に D → E の順に実行されます。C はサーバの応答後なので最後です。

トレース トレース演習 解答
順番何が起きるか

体験 コード体験 1 解答（記録例）
  • Q1: ボタンを押す前は「ボタンを押すとサーバからデータを取得します」と表示されていた。ボタンを押した後、「Webプログラミングチーム」という見出しと、4人のメンバーのリスト（名前と役割）が表示された。
  • Q2: /api/members に直接アクセスすると、JSONデータがそのまま表示された（{"team":"Webプログラミングチーム","members":[...]}）。
  • Q3: ページ全体は再読み込みされなかった。ボタンの下の部分だけが変わった。fetch API を使うとページ全体を再読み込みせずにデータだけ取得して表示を更新できる。
体験 コード体験 2 解答（記録例）
  • Q1: 送信ボタンを押すと、「サーバ応答: ○○さん、メッセージ「xxx」を受け取りました！」と受信時刻が表示された。
  • Q2: ターミナルに 受信: { name: '入力した名前', message: '入力したメッセージ' } が表示された。console.log でサーバ側でもデータを確認できる。
  • Q3: コード体験1（GET）はサーバからデータを取得するだけだったが、コード体験2（POST）はクライアントからサーバにデータを送信し、サーバがそのデータを処理して応答を返した。GETは「もらう」、POSTは「送る」。
体験 コード体験 2 やってみよう解答（空データ送信時）

空の名前・空のメッセージで送信した場合:

  • 画面の表示: 「サーバ応答: さん、メッセージ「」を受け取りました！」と表示される。名前もメッセージも空文字になる。
  • ターミナルの表示: 受信: { name: '', message: '' }（空文字列）

入力欄が空のとき .value は空文字列 ""（undefined ではない）を返します。サーバ側ではエラーにならず、空文字のまま処理されます。実際のアプリでは「名前を入力してください」などのバリデーション（入力チェック）が必要です。
