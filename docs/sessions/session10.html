説明

今回はA-Frame の3D空間と Socket.IO のリアルタイム通信を組み合わせ、複数ユーザが同じ3D空間を共有し、互いのアバターが見える仕組みを作ります。

読み方 ── アバター同期の用語ガイド

3D空間のリアルタイム同期では、ゲーム業界・ネットワーク工学の用語が多数登場します。発音と意味を先に押さえておくと、後のコード読解がぐっと楽になります。

英単語カタカナ読み意味と語源
avatarアバターサンスクリット語「アヴァターラ（化身）」が語源。3D空間内で「自分の化身」として動くキャラクタ。
position / rotationポジション / ローテーション位置 (x, y, z) と回転 (x, y, z)。両方をセットで送らないと「向き」が表現できない。
interpolationインターポレーション（補間）「inter-（間に）＋ polate（差し込む）」＝ 受信した飛び飛びのデータの「間を埋める」処理。
slerpスラープ"Spherical Linear intERPolation"（球面線形補間）の略。回転の補間で使う。
predictionプレディクション（予測）「次にデータが届くまで、今までの動きから先回りで位置を予想する」処理。
dead reckoningデッドレコニング（推測航法）もとは航海用語。最後に受信した位置と速度から、その後の位置を「推測で延長」する技法。
broadcastブロードキャスト「broad（広く）＋ cast（投げる）」＝ 送信者以外の全員に配信。
serializeシリアライズ（直列化）オブジェクトを通信できる文字列（JSON）に変換すること。逆は デシリアライズ。

具体例 ── アバター同期がイメージできる身近な3シーン

場面何が同期されているか今回の仕組みとの対応
オンラインゲーム
（Fortnite / Minecraft）		 各プレイヤーのキャラクタの位置・向き・アクションがサーバ経由で全員に配信される。 socket.emit('move', {position, rotation}) ＋ サーバが broadcast.emit で他全員に転送。これを毎秒10〜60回繰り返す。
ビデオ会議
（Zoom / Teams）		 各参加者の映像枠が画面に並び、退室すると枠が消える。 本資料では映像枠 = アバター。connection でアバター生成、disconnect でアバター削除（クリーンアップ）。
VRChat / cluster
（メタバース）		 同じ仮想空間に集まった人のアバターが3Dで動き、しゃべると口が動く。 本資料の発展形。今回は箱 1個だが、構造はまったく同じ。position/rotation を定期送信 → 他クライアントが再描画。

どれも内部の仕組みは「自分の状態を周期的にサーバへ送る → サーバが他全員に配信 → 各クライアントが受信して再描画」という共通パターンで動いています。

よくある質問 (Q&A) ── アバター同期の疑問

Q1. 100msごとに送信したらカクカクしませんか？ はい、そのままだと毎秒10コマで動くので確かにカクつきます。実用ではクライアント側で補間（interpolation）を入れて、受信した位置の「間」を滑らかにつなぎます。本資料の発展で扱います。
Q2. 通信が遅れたらアバターはどうなる？ 受信が止まると、そのアバターは止まったままになります。これを補うのがプレディクション（予測）とデッドレコニング（推測航法）です。直前の速度から「たぶんこの辺にいるはず」と推測して動かし続けます。
Q3. なぜ broadcast.emit で「送信者以外」に送るの？ 自分の位置データは自分自身ですでに知っているからです。自分のアバターを自分の画面に出す必要もないので、サーバは「送信者本人を除いた全員」にだけ転送するのが効率的です。
Q4. position と rotation の単位は？ A-Frame では position はメートル単位の数値、rotation はオイラー角（度数法、°）です。Three.js 内部ではラジアンと四元数 (Quaternion) に変換されています。
Q5. たくさん人が入ったら通信量は爆発しない？ します。10人なら 10×10 = 100通信／秒、100人なら 100×100 = 10000通信／秒とO(n²)で増えます。実用では 関心領域 (Area of Interest) や 更新頻度の段階化で削減します。

1.1 A-Frame + Socket.IO の組み合わせ

オンラインゲームでは、自分のキャラクターを動かすと、他のプレイヤーの画面でもそのキャラクターが動きます。
今回作るのは、まさにその仕組みのWeb版です。ブラウザで3D空間に入り、他の人のアバターがリアルタイムに動くのが見えます。

第4〜8回で学んだ A-Frame（3D表示）と、第9回で学んだ Socket.IO（リアルタイム通信）を組み合わせます。

技術役割この回での使い方
Node.js + ExpressサーバA-FrameのHTMLファイルを配信し、Socket.IO で接続を管理する
Socket.IOリアルタイム通信各ユーザのカメラ位置・回転データを他の全ユーザに送る
A-Frame3D表示受信したデータをもとに、他ユーザのアバターを3D空間に表示・移動する

1.2 アバターの概念

ビデオ会議ではカメラの映像で相手の存在がわかります。3D空間では映像の代わりにアバター（箱や人型の3Dモデル）で「そこに誰かがいる」ことを表現します。

アバターとは、3D空間内で各ユーザを表すエンティティです。今回は色付きの箱（a-box）をアバターとして使います。

  • サーバに接続したユーザ1人につき、1つのアバター（a-box）が3D空間に生成される
  • 自分のアバターは表示しない（自分のカメラ視点で見ているため）
  • 他のユーザが移動すると、そのアバターの位置・回転がリアルタイムに更新される
  • ユーザが切断すると、そのアバターは3D空間から削除される

1.3 位置・回転データの送信

A-Frame のカメラ（ユーザの視点）の position（位置: x, y, z）と rotation（回転: x, y, z）を定期的にサーバに送信します。

データ内容例
position3D空間内のカメラ座標{x: 0, y: 1.6, z: 3}
rotationカメラの向き（度数）{x: 0, y: -45, z: 0}

これらのデータを 一定間隔（例: 100ms ごと）で socket.emit('move', data) してサーバに送り、サーバが全クライアントに配信します。

なぜ定期送信なのか？

マウスやキーボードの入力イベントは非常に高頻度で発生します。すべてのイベントで送信すると、通信量が膨大になりサーバに負荷がかかります。setInterval で一定間隔に間引くことで、滑らかさと通信効率のバランスを取ります。

1.4 他ユーザのアバター生成と更新

他のユーザから位置データを受信したとき、そのユーザのアバターがまだ存在しなければ新規生成し、存在すれば位置を更新します。

処理を「探す」「無ければ作る」「位置を反映する」の3つの関数に分解すると、各ステップが何をしているか明確になります。

JavaScript（概念コード ── 関数に分割した版）
// ① 既存アバターを ID で探す
function findAvatar(userId) {
  var avatarId = 'avatar-' + userId;
  return document.getElementById(avatarId);
}

// ② アバターが無ければ新規作成して scene に追加
function createAvatar(userId, color) {
  var avatarId = 'avatar-' + userId;
  var avatar = document.createElement('a-box');
  avatar.setAttribute('id', avatarId);
  avatar.setAttribute('color', color);
  scene.appendChild(avatar);
  return avatar;
}

// ③ 位置と回転を反映する
function applyTransform(avatar, position, rotation) {
  avatar.setAttribute('position', position);
  avatar.setAttribute('rotation', rotation);
}

// メイン処理: 「探す → 無ければ作る → 反映する」
socket.on('moved', function(data) {
  var userId = data.id;                       // 中間変数で読みやすく
  var position = data.position;
  var rotation = data.rotation;
  var color = data.color;

  var avatar = findAvatar(userId);

  var isFirstTime = !avatar;
  if (isFirstTime) {
    avatar = createAvatar(userId, color);
  }

  applyTransform(avatar, position, rotation);
});

1.5 ユーザID管理 ── socket.id

Socket.IO では、接続するたびに各クライアントに一意の ID（socket.id）が自動で割り当てられます。

  • socket.id は "xK7-2abc..." のようなランダム文字列
  • この ID を使って「誰の位置データか」「誰のアバターか」を識別する
  • サーバ側でユーザ一覧を管理し、新規接続・切断を追跡する

1.6 ユーザ離脱時のクリーンアップ

ビデオ会議で誰かが退室すると、その人の映像枠が消えます。3D空間でも同じように、ユーザが切断したらそのアバターを削除しないと、動かない幽霊アバターが残り続けます。

サーバの 'disconnect' イベントで、離脱したユーザの ID を全クライアントに通知し、各クライアントが該当のアバターを DOM から削除します。

イベントサーバの処理クライアントの処理
connectionユーザを一覧に追加、色を割り当て（接続完了）
move受信した位置データを他の全員に転送受信したデータでアバターを生成/更新
disconnectユーザを一覧から削除、全員に通知該当アバターをシーンから削除

📖 キーワード

アバター3D空間内で各ユーザを表すエンティティ。ここでは色付きの箱（a-box）を使用。他ユーザの存在と位置を視覚的に示す。
setInterval一定間隔でコールバック関数を実行するJavaScript関数。ここでは100msごとにカメラ位置をサーバに送信する用途で使う。
position / rotation3D空間内のオブジェクトの位置(x,y,z)と回転角度(x,y,z)。カメラのこれらの値を送受信してアバターを同期する。
クリーンアップユーザ切断時にアバターをDOMから削除する処理。これを怠ると動かない「幽霊アバター」が残り続ける。

A-Frameページを配信し、位置データをログに表示する

まずはシンプルな構成で、クライアントのカメラ位置をサーバのコンソールに表示します。

ファイル1: server.js

JavaScript ── server.js
var express = require('express');
var http = require('http');
var { Server } = require('socket.io');

var app = express();
var server = http.createServer(app);
var io = new Server(server);

app.use(express.static('public'));

io.on('connection', function(socket) {
  console.log('接続:', socket.id);

  socket.on('move', function(data) {
    console.log(socket.id, '位置:', data.position);
  });

  socket.on('disconnect', function() {
    console.log('切断:', socket.id);
  });
});

server.listen(3000, function() {
  console.log('http://localhost:3000 でサーバ起動');
});

ファイル2: public フォルダを作成し、その中に index.html を作成

HTML ── public/index.html

    <!-- 空 -->
    <a-sky color="#87CEEB"></a-sky>
  </a-scene>

  <script>
    var socket = io();
    var camera = document.querySelector('[camera]');

    // 100msごとにカメラ位置を送信
    setInterval(function() {
      if (!camera) {
        return;
      }
      var position = camera.getAttribute('position');
      var rotation = camera.getAttribute('rotation');
      var moveData = {
        position: { x: position.x, y: position.y, z: position.z },
        rotation: { x: rotation.x, y: rotation.y, z: rotation.z }
      };
      socket.emit('move', moveData);
    }, 100);
  </script>

例題 例題1 やってみよう 解答

ユーザAが3D空間内で前方（z軸マイナス方向）に1単位だけ移動した場合、各手順で何が起きるかを具体的にトレースした例です。

手順1: ユーザA がWキーを押す → カメラ position が { x:0, y:1.6, z:3 } → { x:0, y:1.6, z:2 } に変化
手順2: setInterval が100ms後に発火 → socket.emit('move', { position:{x:0,y:1.6,z:2}, rotation:{x:0,y:0,z:0} })
手順3: サーバが受信 → socket.broadcast.emit('moved', { id:'xK7-2abc', position:{...}, rotation:{...}, color:'#FF5252' })
手順4: ユーザB が受信 → document.getElementById('avatar-xK7-2abc') が存在 → setAttribute('position', {x:0,y:1.6,z:2})
手順5: A-Frame が次フレームで再描画 → 赤い箱が z=3 から z=2 に移動して見える

重要ポイント: サーバは broadcast.emit を使うことで「送信者本人以外の全員」に転送します。これにより、ユーザA は自分の箱を自分の視界に作る必要がなく、カメラに重なって見えなくなる問題を回避できます。また、ユーザB側ではアバター ID（avatar- + socket.id）で一意性を担保し、初回受信時のみ createElement で生成、それ以降は setAttribute で位置を更新するだけ。これにより毎秒10回（100msごと）の更新でも DOM 操作が最小限に抑えられます。
